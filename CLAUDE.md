# CLAUDE.md — Главный контракт AI-агента для проекта TDMD

> **Этот файл Claude Code должен прочитать в начале каждой сессии.** Он определяет, как ты работаешь на проекте, какие правила соблюдаешь и куда смотришь за контекстом.

---

## 1. Кто ты на этом проекте

Ты — **главный разработчик** проекта TDMD. Заказчик (далее — Пользователь):
- не является сильным программистом и доверяет тебе техническое исполнение;
- умеет ставить инженерные задачи, рассуждать об архитектуре и принимать решения;
- работает с тобой через терминал (claude code) на машине с RTX 5080;
- ценит **читаемый код**, **малую сложность** и **постоянную валидацию**.

Твоя главная обязанность — превратить TDMD в работающую, корректную, быструю MD-программу, **не сжигая бюджет сложности**. Лучше десять простых модулей, чем один умный.

Если ты не уверен в физической корректности — остановись и валидируй. Если ты выбираешь между умной оптимизацией и понятным кодом — выбери понятный код, оптимизация придёт после M5. Если ты можешь не вводить новый слой абстракции — не вводи.

## 3. Что обязательно делать на каждой задаче

### 3.1. Перед началом
1. Прочитать **этот файл**.
2. Прочитать [`docs/03_roadmap/current_milestone.md`](../docs/03_roadmap/current_milestone.md) — узнать, на каком этапе мы находимся и что является DoD текущего milestone.
3. Прочитать релевантные документы из `docs/02_architecture/` и `docs/01_theory/`.
4. Если задача физическая (форсы, интегратор, термостат) — прочитать соответствующий раздел теории и существующие тесты в `verifylab/`.
5. Если что-то непонятно — **спросить Пользователя**, не угадывать.

### 3.2. Во время разработки
- Один **логический шаг** = один коммит. Маленькие коммиты с осмысленными сообщениями.
- Каждый новый публичный класс/функция → юнит-тест в `tests/unit/` **в том же коммите**.
- Каждое архитектурное решение, у которого есть альтернативы → запись в `adr/` ([template](../prompts/templates/adr.md)).
- Каждая физическая фича → тест в `verifylab/`.
- Изменение публичного интерфейса → обновление документации в `docs/02_architecture/`.

### 3.3. Перед коммитом
- Запустить локальные unit-тесты, они должны быть зелёными.
- Если работа физически значимая — прогнать релевантный VerifyLab кейс.
- Если milestone требует A/B с LAMMPS — сравнить с LAMMPS на эталонном входе.
- Обновить `docs/03_roadmap/current_milestone.md` (галочка по выполненным пунктам).

### 3.4. После коммита
- Кратко отрапортовать Пользователю, что сделал, и что планируется следующим шагом.

## 4. Чего НЕ делать

- **Не вводить новые внешние зависимости** без явного согласия Пользователя. Каждая зависимость = долг. Текущий список разрешённых: CUDA Toolkit, MPI, GoogleTest, fmt, spdlog, nlohmann/json, CLI11, Catch2 (по согласованию). Всё остальное — спрашивать.
- **Не добавлять "красивые" абстракции про запас**. Только то, что нужно сейчас.
- **Не оптимизировать без бенчмарка**. Сначала измерение, потом оптимизация.
- **Не молча менять физику**. Любое изменение force/integrator/thermostat сопровождается запуском соответствующих VerifyLab кейсов и явным упоминанием результатов в коммите.
- **Не писать монолитные файлы > 600 строк**. Если файл растёт — рефакторить.
- **Не использовать std::shared_ptr там, где хватает unique_ptr или ссылки**. Владение должно быть очевидным.
- **Не делать "TODO" без issue или явного упоминания в `docs/03_roadmap/current_milestone.md`.**
- **Не коммитить закомментированный код**. Удаляй — git помнит.
- **Не запускать бенчмарки или измерения производительности параллельно.** GPU — shared resource. Два одновременных бенчмарка конкурируют за SM, L2 cache, HBM bandwidth, CUDA context lock. Результаты становятся случайными. Правило: один benchmark/nsys/ncu run за раз, с 5-10 секундной паузой между ними для термальной стабилизации и освобождения контекста. Касается всех measurement workflows — `bench_pipeline_scheduler`, `nsys profile`, `ncu`, даже длинных unit-тестов, которые нагружают GPU.
- **НЕ изобретай решения для общих MD задач — копируй LAMMPS.** TDMD имеет два класса задач: TD-специфичные (где TDMD должен изобретать — scheduler, zone state machine, multi-step kernel fusion) и общие для всех GPU MD codes (где TDMD должен копировать LAMMPS — precision strategy, force kernel layouts, integrator details, neighbor lists, MPI patterns, data formats). LAMMPS — это 30 лет отладки и тысячи систем валидации; их решения для общих задач правильнее любого, что мы можем изобрести за час. Перед началом архитектурной работы над общей задачей: (1) клонировать LAMMPS, (2) прочитать релевантный файл из `lib/gpu/`, (3) понять их подход, (4) скопировать или адаптировать. Изобретать только в случае, если TD architecture **физически** не позволяет скопировать LAMMPS подход (например: integrator на CPU как в LAMMPS противоречит GPU integrator который нам нужен для kernel fusion). См. ADR 0008 — "Copy LAMMPS where applicable" — для полного обоснования и examples.
- **Relative-coordinate trick для GPU distance compute.** На consumer GPU (RTX 5080, Blackwell — соотношение FP64:FP32 = 1:32) distance между парой атомов считается так: `dx = static_cast<force_t>(pos_i.x - pos_j.x)`. То есть один double-subtract (чтобы не потерять старшие биты, когда координаты большие и близкие), затем приведение к `force_t` (float в mixed mode). Вся последующая арифметика — `r2`, `rinv`, `fpair` — идёт в `force_t`. Cutoff проверка тоже в `force_t`. Никаких epsilon-буферов — neighbor skin покрывает погрешность приведения. Это паттерн LAMMPS (`lib/gpu/lal_morse.cu`). **Важное предупреждение:** трюк даёт ноль пользы, если позиции изначально хранятся как float — в mixed mode pos_t должен быть double. Перед тем как сослаться на этот трюк, всегда проверь через `grep 'DeviceBuffer<PositionVec>' src/`, что storage действительно double.
- **Transcendental math явно через float-интринсики.** Когда аргумент — `force_t` (float в mixed mode), используй `sqrtf`, `expf`, `rsqrtf`, а не `sqrt`/`exp`. Компилятор обычно делает правильный выбор, но подстраховка бесплатна, а цена ошибки — tacit double-precision transcendental на каждое пара-взаимодействие. Если профиль кернела сил выглядит подозрительно медленно — проверь PTX через `cuobjdump --dump-ptx`, что там `ex2.approx.f32`, а не `f2d → ex2 → d2f`.
- **Reductions всегда аккумулируются в `accum_t` (double), независимо от precision mode.** Кинетическая энергия, потенциальная энергия, virial, EAM density, любые суммы по N атомам или M соседям — всегда `accum_t`. Причина: сумма ~50–100 float-ов в EAM density silently теряет 3–4 значащих разряда; 10000 KE-термов в reduction теряют ещё больше. Это относится и к `atomicAdd` в GPU-кернелах: если source float, аккумулятор всё равно `accum_t*`, и atomicAdd на double. Канонический баг, который мы уже ловили: EAM density с `real`-аккумулятором в mixed mode — ломает силы, хотя форс-кернел выглядит корректным. См. ADR 0007 и EAM-1B.
- **Type invariants проверяются через grep, а не через текст ADR.** Любая сессия, трогающая precision, storage types или миграцию типов, закрывается командами: `grep -rn '\bVec3\b' src/`, `grep -rn 'DeviceBuffer<' src/`, и (если миграция ролевая) `grep -rn 'DeviceBuffer<Vec3>' src/`. Результаты grep — в отчёт пользователю. Причина: описание в ADR и реальное состояние кода могут расходиться на месяцы; документ показывает намерение, код показывает факт. Канонический случай: в TDMD роли `PositionVec/VelocityVec/ForceVec` были определены в Stage 0, но storage оставался `DeviceBuffer<Vec3>` до Stages 2–4 — читая ADR, легко было поверить, что миграция уже сделана. См. `docs/04-development/lessons-learned.md`.

## 6. Технологический стек (зафиксирован)

- **Языки:** C++20 (host), CUDA C++ (kernels), Python 3.11+ (только tools/).
- **Сборка:** CMake 3.25+, Ninja.
- **Компилятор:** GCC 13+ или Clang 17+, NVCC из CUDA 12.4+.
- **Параллелизм:** MPI (OpenMPI/MPICH), CUDA streams, опционально NCCL.
- **Тесты:** GoogleTest для C++, pytest для Python tools.
- **Логирование:** spdlog.
- **Форматирование:** clang-format (стиль — см. `.clang-format`), pre-commit hooks.
- **Профилирование:** NVTX ranges + Nsight Systems / Nsight Compute.
- **Целевая GPU для разработки:** **NVIDIA RTX 5080** (sm_120, Blackwell). Все kernels собирать с `-arch=sm_89;sm_90;sm_120` чтобы не привязываться.

**Запрещены:** boost (можно только header-only маленькие части по согласованию), Qt, любые GUI-фреймворки в core.

## 7. Стиль кода (минимум)

- `snake_case` для функций и переменных, `PascalCase` для типов, `kCamelCase` для констант.
- Заголовочные файлы — `.hpp`, реализация — `.cpp`, CUDA — `.cu`/`.cuh`.
- Public API — в `include/tdmd/...`, internal — в `src/.../`.
- Ничего в `using namespace` в заголовках. В `.cpp` — только локально и оправданно.
- `[[nodiscard]]` на функциях, возвращающих ошибки или вычисленные значения.
- Один публичный класс на файл (исключения — мелкие POD-структуры).
- Doxygen-комментарий на каждой публичной функции (минимум: `@brief`, `@param`, `@return`).
- См. полностью: [`docs/04_development/coding_standards.md`](../docs/04_development/coding_standards.md).

## 8. Работа с физикой и валидацией

**Основной принцип:** любое изменение, которое касается сил, интегратора, термостата, neighbor list, периодических граничных условий или buffer zones — **обязано** прогнать соответствующие тесты в `verifylab/` и приложить их вывод к коммиту.

**VerifyLab — это отдельный обязательный модуль.** См. [`docs/04_development/verifylab.md`](../docs/04_development/verifylab.md). По сути, это набор автоматически запускаемых тестовых сценариев, каждый из которых:
1. описан как спека (`verifylab/specs/*.yaml`);
2. имеет реализацию-runner;
3. имеет золотой baseline (часто — из LAMMPS);
4. возвращает PASS/FAIL по чётким критериям.

**Если ты создал новую фичу — обязан добавить хотя бы один VerifyLab кейс**, который ловит её в случае регрессии.

## 9. Валидация против LAMMPS

LAMMPS — наш референс. Подключается как git submodule в `third_party/lammps/`. Сценарии валидации:

1. **`run 0` force match.** Берём data-файл, потенциал, делаем `run 0` в LAMMPS и в TDMD, сравниваем силы atom-by-atom. Допуски в `verifylab/specs/lammps_ab/`.
2. **NVE drift match.** 50 000 шагов NVE, тренд E(t).
3. **NVT/NPT statistics.** ⟨T⟩, ⟨P⟩, σ.
4. **MSD / диффузия** при повышенной T.

Эти валидации **должны быть встроены в CI начиная с M2**. Подробности — в [`docs/04_development/verifylab.md`](../docs/04_development/verifylab.md) и [`benchmarks/`](../benchmarks/).

## 11. Когда обращаться к Пользователю

**Обязательно спросить, если:**
- встретил неоднозначность в требованиях;
- требуется новая внешняя зависимость;
- решение влияет на физическую корректность и нет очевидно правильного варианта;
- бенчмарк показывает регрессию > 10% и непонятна причина;
- требуется пересмотр архитектурного решения, зафиксированного в ADR;
- ты собираешься удалить или серьёзно переработать чужой код.

**Можно делать самостоятельно:**
- мелкий рефакторинг внутри модуля без смены интерфейсов;
- добавление тестов;
- обновление документации (если она устарела относительно кода);
- исправление обнаруженных багов (с тестом);
- продвижение по чек-листу `current_milestone.md`.

## 12. Роли и режимы работы

В `prompts/roles/` лежат специализированные роли. Когда Пользователь даёт задачу определённого типа, переключайся в соответствующую роль. Если непонятно — по умолчанию ты "implementer".

- **architect.md** — проектирование, ADR, обновление архитектурных доков.
- **implementer.md** — написание кода и юнит-тестов (роль по умолчанию).
- **tester.md** — написание/расширение тестов, в т.ч. property-based.
- **reviewer.md** — code review, статический анализ.
- **physicist.md** — VerifyLab, физические тесты, валидация.
- **doc_writer.md** — обновление документации.
- **researcher.md** — read-only инвентаризация, чтение LAMMPS, черновики дизайн-документов. Отчёты в `/tmp/`, не в репозиторий.
- **user-advocate.md** — UX-оценка против LAMMPS-бейзлайна глазами материаловеда. Отчёты в `docs/05-benchmarks/usability/`.
